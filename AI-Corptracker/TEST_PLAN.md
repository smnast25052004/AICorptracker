# Тест-план: AI CorpTracker

Документ описывает, **что и в каком порядке проверять** при приёмочном и регрессионном тестировании. Предполагается окружение: Docker Compose поднят, при необходимости выполнен `seed`.

---

## 1. Предусловия

| ID | Условие | Как проверить |
|----|---------|----------------|
| P1 | Установлены Docker и `docker compose` | `docker compose version` |
| P2 | Сервисы запущены | `docker compose ps` — все нужные контейнеры `Up` |
| P3 | PostgreSQL отвечает | контейнер `postgres` в состоянии healthy |
| P4 | Kafka healthy (если используется полный стек) | контейнер `kafka` healthy |
| P5 | (Опционально) Загружены демо-данные | `docker compose exec api python -m seed.seed_data` |

---

## 2. Дымовые тесты (Smoke)

| ID | Сценарий | Ожидаемый результат |
|----|----------|---------------------|
| S1 | `GET http://localhost:8000/health` | HTTP 200, JSON с `"status":"healthy"` |
| S2 | Открыть `http://localhost:8000/docs` | Загружается Swagger UI, перечислены теги API |
| S3 | `GET /api/dashboard/summary` | HTTP 200, JSON без ошибки парсинга |
| S4 | Открыть `http://localhost:8501` | Streamlit загружается, страницы навигации открываются |
| S5 | После `docker compose restart` | S1–S3 снова проходят в течение 1–2 минут |

---

## 3. API — REST (функциональные)

Базовый URL: `http://localhost:8000`. Проверять код ответа, структуру JSON и отсутствие 500 на валидных запросах.

### 3.1 Dashboard

| ID | Метод и путь | Проверка |
|----|--------------|----------|
| A1 | `GET /api/dashboard/summary` | Есть поля сводки (цели, риски, рекомендации — по схеме ответа) |

### 3.2 Цели (Goals)

| ID | Метод и путь | Проверка |
|----|--------------|----------|
| G1 | `GET /api/goals/` | Список целей; после seed — непустой |
| G2 | `GET /api/goals/{id}` | Валидный UUID из G1 — детали цели, проекты, задачи |
| G3 | `GET /api/goals/00000000-0000-0000-0000-000000000000` | Ожидаемое поведение: 404 или объект с `error` (как задумано в коде) |
| G4 | `POST /api/goals/` | Создание цели по телу из Swagger — 200/201, запись в БД |
| G5 | `PUT /api/goals/{id}` | Обновление полей — ответ успешный, повторный GET отражает изменения |
| G6 | `DELETE /api/goals/{id}` | Удаление тестовой цели (созданной в G4) |

### 3.3 Риски

| ID | Метод и путь | Проверка |
|----|--------------|----------|
| R1 | `GET /api/risks/` | Список оценок рисков |
| R2 | `GET /api/risks/recommendations` | Список рекомендаций |

### 3.4 Поиск

| ID | Метод и путь | Проверка |
|----|--------------|----------|
| Q1 | `GET /api/search/semantic?q=лицензия` | HTTP 200, поле `results` (массив), `count` |
| Q2 | `GET /api/search/semantic?q=` | Поведение при пустом запросе — ошибка валидации 422 или явное сообщение |

### 3.5 Анализ

| ID | Метод и путь | Проверка |
|----|--------------|----------|
| N1 | `POST /api/analysis/run` | HTTP 200, `status`/`goals_analyzed` без ошибки |
| N2 | `GET /api/analysis/notifications` | JSON с уведомлениями (может быть пустой список) |

### 3.6 Сотрудники, проекты, задачи, документы

| ID | Метод и путь | Проверка |
|----|--------------|----------|
| E1 | `GET /api/employees/` | Список |
| E2 | `GET /api/employees/{id}` | Детали по существующему id |
| P1 | `GET /api/projects/` | Список |
| P2 | `GET /api/projects/{id}` | Детали |
| T1 | `GET /api/tasks/` | Список |
| T2 | `GET /api/tasks/{id}` | Детали |
| D1 | `GET /api/documents/` | Список |
| D2 | `GET /api/documents/{id}` | Детали |

Для **POST/PUT/DELETE** по employees, projects, tasks, documents — пройти сценарий «создать → изменить → удалить» по примеру тел из Swagger (как в G4–G6).

## 4. Интеграция и фоновые сервисы

| ID | Сценарий | Ожидаемый результат |
|----|----------|---------------------|
| I1 | Логи `fetcher`: `docker compose logs fetcher --tail 50` | Нет постоянных traceback; периодические циклы опроса |
| I2 | Логи `processor` | Нет фатальных ошибок при обработке событий |
| I3 | Логи `decision_engine` | Периодический цикл анализа без падения контейнера |
| I4 | После `POST /api/analysis/run` | В БД обновляются/появляются записи рисков/рекомендаций (проверка через API или SQL) |

## 8. Негативные и граничные случаи

| ID | Сценарий | Ожидаемый результат |
|----|----------|---------------------|
| N1 | API недоступен (контейнер api остановлен) | Streamlit показывает ошибку или пустое состояние, без падения процесса |
| N2 | Неверный UUID в пути | 404 или согласованное сообщение об ошибке |
| N3 | Дублирующий POST с тем же бизнес-ключом | Поведение по правилам API (409/422 или разрешённый дубликат — зафиксировать фактическое) |

## 11. Критерии приёмки (рекомендуемые)

- Все пункты **раздела 2 (Smoke)** и **3.1–3.5** проходят на чистом окружении после seed.
- Контейнеры **api**, **postgres**, **kafka** (если используется) — стабильно `Up` 30+ минут.
- Нет необработанных **500** на стандартных GET из раздела 3 при валидных данных.
